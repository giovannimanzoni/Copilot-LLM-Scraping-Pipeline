overview redis repo scanner reset
reset operations
reset all data
Wipes all Redis state and re-seeds from scratch. Use this to start a completely fresh run.
  • Deletes all repos (pending, assigned, done) and re-seeds from input_data/repos_found.txt
  • Deletes all stack tasks (pending, assigned, done, problematic) and re-seeds
  • Clears all worker stats and heartbeats
  • Unchecks all dashboard checkboxes (auto-advance, phase enables, shared-fs)
reset stack tasks
Wipes all stack state (pending, assigned, done, problematic) and re-seeds. Use this to restart stack tasks.
  • Repos are not affected
  • Worker stats are not affected
reset problematic stack tasks
Moves all problematic stack tasks back into the pending queue so they can be retried. Does not touch repos or worker stats.
retry problematic repos
Retries all problematic repos one at a time — dispatches one repo, waits for the worker to report it done or failed, then dispatches the next. Does not touch stack tasks or worker stats.
reset worker names
Clears the registered name for all workers. Names will be re-registered automatically on next worker startup. All other worker stats, repos, and stack data are not affected.
reset by phase
phase 3 — merge
Wipes all merge state (ready node IDs, status, assigned nodes, done nodes, shard progress), stops the fleet, and deletes partial/final dataset files from disk. Stack and repos are not affected.
phase 2 — stack
Wipes all stack data (pending, assigned, done, problematic) and re-seeds from scratch. Repos are not affected.
phase 1 — repos
Wipes all repo data (pending, assigned, done) and re-seeds from input_data/repos_found.txt. Stack is not affected.
⚠
reset all data?
This will permanently delete all scraped repos, stack tasks, and worker stats from Redis, then re-seed from defaults.

This action cannot be undone.
↺
reset stack tasks?
In-flight and problematic stack state will be cleared and re-seeded. Workers that already completed their shard are skipped.

Repos, worker stats, and already-done stack tasks are not affected.
↺
retry problematic repos?
Problematic repos will be retried one at a time. The coordinator dispatches one repo, waits for the worker to report it done or failed, then dispatches the next.

Worker stats and all other data are not affected.
↺
retry problematic tasks?
All problematic stack tasks will be moved back to the pending queue and retried by workers.

Worker stats and all other data are not affected.
⚠
reset merge phase?
This will delete all merge data — including node status, shard progress, and done records — stop the fleet, and delete any partial/final dataset files from disk.

Stack and repos are not affected. This action cannot be undone.
⚠
reset stack phase?
This will delete all stack data — including completed batches — and re-seed from scratch.

Repos are not affected. This action cannot be undone.
↺
reset repos phase?
All repo data (pending, assigned, done) will be wiped and re-seeded from input_data/repos_found.txt.

Stack is not affected.
↺
reset worker names?
All registered worker names will be cleared from Redis. Workers will re-register their names automatically on next startup.

All other data is not affected.
